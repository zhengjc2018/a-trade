# A-Trade 全量问题修复设计

## 1. 目标

本次修复覆盖 2026-07-19 全面审查报告列出的 P0-P3 问题。完成后，项目必须满足以下结果：

1. 公开仓库不再包含真实持仓、成本、数量、买入日期、私有群标识或具体 VPS 地址。
2. 行情字段、价格、成交量、成交额、估值和时间单位具有明确且经过测试的数据契约。
3. 日线缓存能够检测陈旧数据并安全刷新，离线回测不会被实时快照污染。
4. 回测使用现金和股份守恒账本，不存在同 K 线成交、重复计股或卖出未解锁股份。
5. 通知只有在平台确认成功后才提交去重状态，失败可观察且可重试。
6. 新环境按仓库声明的依赖即可运行、测试和部署，不依赖系统全局包。
7. 主要入口、数据契约、账本不变量和部署脚本都能被自动验证。

## 2. 约束

- 保留现有 `scripts/run_backtest.py`、`scripts/screen.py` 和 `scripts/run_scheduler.py` 入口。
- 尽量保留现有公开类名；确需改变内部模型时提供清晰迁移边界。
- 单元测试不得发送真实 QQ 消息或依赖实时交易接口。
- 实时接口只用于人工契约核对，自动测试使用固定响应样例。
- 不自动提交、推送、改写 Git 历史或轮换外部平台凭据。
- 不引入数据库服务、消息队列或容器等当前规模不需要的基础设施。
- 修复顺序以正确性和安全性优先，不保留已确认错误的历史行为。

## 3. 方案选择

采用兼容式分阶段重构：保留外部入口和主要模块边界，在内部逐步替换错误实现，并在每个阶段建立回归测试。

未采用整体重写，因为它会同时改变所有调用关系，增加无法定位回归的风险。未采用局部补丁，因为回测账本、配置安全和通知送达问题需要新的明确状态模型，表面修补无法保证不变量。

## 4. 配置与敏感信息设计

### 4.1 文件布局

- `config/holdings.example.json`：可提交的脱敏持仓示例。
- `config/monitor.example.json`：可提交的脱敏监控示例。
- `config/holdings.local.json`：真实持仓，本地保留并加入 `.gitignore`。
- `config/monitor.local.json`：真实监控配置，本地保留并加入 `.gitignore`。
- `atrade/config.py`：唯一配置解析、路径解析和校验入口。

加载优先级为显式环境变量路径、`*.local.json`。缺少真实配置时返回明确错误和复制示例的命令，不自动把示例当作真实持仓运行。

### 4.2 校验规则

- 股票代码必须为 6 位数字。
- 成本价必须大于 0，数量必须为正整数。
- 扫描间隔和数据长度必须为正整数。
- K 线周期仅允许项目支持的固定集合。
- 未知顶层字段、错误 JSON 和缺失必需字段必须阻止启动，不静默回退为空列表。

### 4.3 部署信息

文档和脚本示例统一使用 `deploy@your-vps`、`/opt/a-trade` 等占位符。服务使用独立用户，SSH 只允许密钥认证，重启权限限制到单一 systemd 服务。

## 5. 行情与缓存设计

### 5.1 全市场快照契约

东财 `fltt=2` 响应使用实际价格：

- `f2` 最新价
- `f3` 涨跌幅
- `f4` 涨跌额
- `f5` 成交量（手）
- `f6` 成交额（元）
- `f7` 振幅
- `f8` 换手率
- `f9` 动态市盈率
- `f10` 量比
- `f12` 股票代码
- `f14` 名称
- `f15` 最高价
- `f16` 最低价
- `f20` 总市值（元）
- `f23` 市净率

包内统一使用实际价格和人民币元，不再存储乘 100 的价格。筛选、表格和测试都消费同一数据契约。

### 5.2 历史 K 线

- 新浪 K 线成交量按股处理，估算成交额为 `close * volume`。
- 日线日期保存为 `YYYY-MM-DD`。
- 分钟线时间保存为 `YYYY-MM-DD HH:MM:SS`，不得截断为日期。
- 派生字段计算前先校验 OHLCV 列、数值类型、价格关系和非负成交量。

### 5.3 缓存刷新

日线缓存使用最后数据日期判断新鲜度。上海时区收盘前的当天不要求已有完整日线；收盘后要求更新到最近工作日。缓存陈旧时拉取数据并 upsert，拉取失败时允许返回旧缓存，但结果携带可观察的 warning。

实时基本面只附加到当前返回的 DataFrame，不写回某个历史交易日。离线回测默认 `use_snapshot=False`。

### 5.4 TTM EPS

最新报表为 Q1、中报或 Q3 时，计算公式为上一年年报累计 EPS，加本年当前周期累计 EPS，减上一年同周期累计 EPS。年报直接使用年报 EPS；缺少同期数据时明确标记降级来源。

## 6. 回测账本设计

### 6.1 状态模型

回测维护以下独立状态：

- `target_quantity`：策略希望长期保持的底仓数量。
- `settled_quantity`：当前可卖股份。
- `locked_quantity`：当日买入、下一交易日才可卖的股份。
- `trading_cash`：所有 T 交易现金流，买入为负，卖出为正，费用立即扣除。
- `cycle_start_cash`：一轮正向或反向做 T 开始前的累计现金，用于计算单轮盈亏。

总股份始终等于 `settled_quantity + locked_quantity`。正向做 T 时总股份高于目标，反向做 T 时总股份低于目标，平仓后恢复到目标。

### 6.2 成交规则

- 第 N 根 K 线收盘后确认的信号最早在第 N+1 根 K 线开盘成交。
- 同一时刻信号优先级为止损、卖出、买入，每根 K 线最多执行一个新方向动作。
- 买入增加锁定股份并减少现金。
- 卖出只能减少已结算股份并增加现金。
- 新交易日开始时，前一交易日锁定股份全部转为可卖股份。
- `force_eod_close=True` 时，在当天最后一根 K 线按可配置的收盘成交模型把总股份恢复到目标。

### 6.3 正向与反向做 T

- 总股份等于目标时收到 BUY：买入一个 T 仓，开启正向周期。
- 总股份高于目标时收到 SELL 或到达 EOD：卖出可卖底仓，将总股份恢复到目标。
- 总股份等于目标时收到 SELL：卖出一个 T 仓，开启反向周期。
- 总股份低于目标时收到 BUY 或到达 EOD：买回缺口，将总股份恢复到目标。
- 已有未平周期时不叠加同方向仓位。

### 6.4 估值与费用

- 策略净值为 `总股份 * 当前价 + trading_cash`。
- T 净盈亏为 `trading_cash + (总股份 - 目标股份) * 当前价`。
- 有效成本为 `原始成本 - T 净盈亏 / 目标股份`。
- 佣金支持最低收费，卖出支持印花税，买卖支持过户费和滑点。
- 年化指标只有在区间和分母有效时计算，不允许除零、复数或无限值。

### 6.5 必须成立的不变量

- 卖出数量不得超过 `settled_quantity`。
- `settled_quantity` 和 `locked_quantity` 不得为负。
- 平仓周期结束后总股份等于目标股份。
- 每笔成交后的现金变化必须等于成交金额和费用之和。
- 没有交易时策略净值变化必须与死拿持仓一致。
- 正收益必须降低有效成本，负收益必须提高有效成本。

## 7. 通知与调度设计

### 7.1 通知端口

保留一个面向业务的通知接口，提供文本和 Markdown 发送。REST 和 botpy 作为内部适配器，不允许 CLI、调度器和业务模块各自定义不同方法名。

Markdown 按 UTF-8 字节限制分段，优先在段落边界拆分。发送结果包含成功状态、消息 ID、错误类型和是否可重试。

### 7.2 去重和送达

监控扫描只生成候选告警，不立即写入已发送状态。通知平台确认成功后，再把带 TTL 的告警 key 写入状态文件。失败时保留为未发送，允许下一次任务重试。

### 7.3 生命周期

调度器使用 READY 事件等待 bot 客户端连接。超时或线程异常时启动失败，不记录虚假成功。停止顺序为停止接收新任务、等待已提交 Future、关闭客户端、关闭事件循环并 join 线程。

## 8. 部署与工程化设计

### 8.1 依赖

- `requirements.txt` 固定经过验证的运行依赖并包含 `akshare`。
- `requirements-dev.txt` 固定 pytest、Ruff、coverage 和依赖审计工具。
- `pyproject.toml` 声明 Python 版本、pytest 路径和静态检查配置。

### 8.2 CI

GitHub Actions 在隔离环境中执行：安装依赖、核心导入、pytest、compileall、Ruff、Shell 语法和 CLI 帮助检查。单元测试不读取本地 `.env`，不发送外部消息。

### 8.3 部署

服务以 `atrade` 用户运行。部署先在新 release 目录安装锁定依赖并执行核心导入、编译和测试，再切换 `current` 软链接并重启。健康检查失败时恢复上一 release。post-receive 不升级 pip，不以 root 运行应用。

## 9. 兼容与迁移

- `run_backtest.py`、`screen.py`、`run_scheduler.py` 命令保留。
- `--portfolio` 改为使用统一配置加载器。
- 原 `config/holdings.json` 和 `config/monitor.json` 的真实内容在本地迁移到 `*.local.json` 后，从可提交文件中删除。
- 旧 T 监控状态文件在读取时兼容一次，随后写成带 `sent` 映射和时间戳的新格式。
- 回测结果保留已有核心字段，同时增加最终数量、交易现金和峰值资金占用，旧字段语义在报告中明确。

## 10. 测试策略

测试分为五层：

1. 配置测试：路径优先级、脱敏文件、字段校验和错误信息。
2. 数据契约测试：固定东财/新浪响应、单位、分钟时间、缓存刷新和 TTM 周期。
3. 账本测试：下一根 K 线成交、正向/反向周期、T+1、费用、守恒和期末盯市。
4. 通知测试：UTF-8 分段、成功后提交、失败重试、旧状态迁移和 READY 超时。
5. CLI/部署测试：帮助命令、核心导入、Shell 语法、systemd 模板和 CI 全流程。

所有已有测试继续运行；与已确认错误行为绑定的测试改为正确契约，而不是保留错误兼容。

## 11. 完成标准

- 全部 P0-P3 条目在审查报告中标记为已修复或给出必须由用户执行的外部操作。
- 新增测试能够在修复前复现关键错误，并在修复后通过。
- 完整 pytest、compileall、Ruff、Shell、plist 和 CLI 检查通过。
- Git 工作区中不存在真实持仓或具体 VPS 地址的可提交副本。
- 回测至少通过现金守恒、股份守恒、下一根 K 线成交和正反向周期测试。
- 通知至少通过成功提交、失败不提交和 UTF-8 字节限制测试。
- TODO 和 STATUS 与实际进展一致，并记录最终验证命令和结果。
